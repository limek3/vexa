# ClickBook data source of truth

Решение: **канонический источник данных для MVP — `sloty_workspaces.data` JSON**.

Нормализованные таблицы (`sloty_services`, `sloty_availability_days`, `sloty_message_templates`, `sloty_bookings`, `sloty_chat_threads` и т.д.) используются как:

1. read-model / индекс для публичной записи, аналитики и быстрых API-запросов;
2. backward-compatible fallback, если в JSON-секции ещё нет данных;
3. подготовка к будущему разделению доменов на отдельные таблицы.

Правило для бизнес-логики:

- если JSON-секция `services`, `availability`, `templates` или `bookings` непустая — она имеет приоритет;
- нормализованные таблицы можно использовать только как fallback или производное представление;
- запись через dashboard сначала обновляет `sloty_workspaces.data`, затем best-effort синхронизирует нормализованные таблицы;
- сбой синхронизации read-model не должен ломать пользовательское сохранение JSON-данных.

Почему так:

- dashboard и Mini App уже редактируют JSON-секции;
- это минимально рискованно для текущего MVP;
- публичная страница перестаёт получать устаревшие слоты из таблиц, если JSON уже содержит свежие изменения;
- переход на полностью нормализованную модель можно сделать отдельной миграцией без смешивания приоритетов.

Перед будущей миграцией на SQL-first нужно будет:

1. перенести все dashboard-edit flows на нормализованные таблицы;
2. добавить транзакционную синхронизацию/rollback;
3. убрать JSON fallback из публичной записи и бронирований;
4. оставить JSON только для редких settings/feature flags.
